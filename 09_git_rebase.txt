what is difference between pull, fetch and rebase ??

These are common Git commands for managing changes between local and remote repositories. I'll explain each one, their purposes, and key differences. Note that pull and fetch are primarily for syncing with remotes, while rebase is a way to integrate changes (often used in conjunction with them).



Fetch

Command: git fetch <remote>
What it does: Downloads the latest commits, branches, and tags from a remote repository (e.g., origin) to your local repository, but does not merge them into your current branch. It updates your local references (like origin/main) without affecting your working files or current branch.
Key behavior: Safe and non-destructive; it's essentially a "preview" of remote changes. You can inspect differences (e.g., via git log origin/main) before deciding to merge or rebase.
When to use: When you want to see what's new on the remote without committing to changes in your local work. Common in team workflows to check for updates.



Pull

Command: git pull <remote> <branch>
What it does: A combination of git fetch followed by git merge. It fetches remote changes and automatically merges them into your current branch.
Key behavior: Can create merge commits if there are diverging changes, which preserves history but might clutter the commit graph with extra "merge" entries. If conflicts arise, you'll need to resolve them manually.
When to use: For quick updates when you're okay with merge commits and want to integrate remote changes immediately. Default behavior in many workflows, but can lead to a non-linear history.



Rebase

Command: git rebase <base> (often used as git rebase <remote>/<branch> after fetching)
What it does: Moves your local commits to a new base commit, effectively "replaying" them on top of the updated branch (e.g., after fetching remote changes). This creates a linear history without merge commits.
Key behavior: Rewrites commit history, which can make it cleaner but is dangerous if you've already pushed those commits (it can cause issues for collaborators). Never rebase shared branches. Often paired with fetch/pull for cleaner integration.
When to use: To maintain a clean, linear commit history (e.g., before merging a feature branch). Use git pull --rebase to fetch and rebase in one step instead of merging.
